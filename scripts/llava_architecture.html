Why 336×336? The original CLIP ViT-L/14 uses 224×224. The "-336" variant was fine-tuned at higher resolution for better detail. 336 / 14 = 24, giving a 24×24 patch grid = 576 patches.
Stage 2 — Patch Embedding (Image → Token Sequence)
A single Conv2d turns the image into a flat sequence of patch vectors
336×336 image → 24×24 grid (showing 6×6 subset)

Conv2d Patch Embedding

Conv2d( in_channels = 3, // RGB out_channels = 1024, // hidden_size kernel_size = 14, // patch_size stride = 14 // non-overlapping )

Each 14×14 pixel region is multiplied by a learned weight matrix and collapsed into a single 1024-dim vector. No overlap between patches.

// Pseudocode: input: (1, 3, 336, 336) ↓ Conv2d(kernel=14, stride=14) output: (1, 1024, 24, 24) ↓ flatten spatial dims (1, 576, 1024) // ↑ ↑ ↑ // batch seq embed
Key insight: This Conv2d is equivalent to slicing the image into 576 non-overlapping 14×14 patches, flattening each to 588 values (14×14×3), and multiplying by a (588 → 1024) linear layer. The Conv2d formulation is just more efficient on GPU.

[CLS] token: A learnable vector prepended at position 0. After passing through all transformer layers, it aggregates global image information via attention. CLIP uses it for image-text matching. LLaVA drops it — it uses the patch tokens instead.
Position embeddings: Without these, the transformer has no notion of spatial arrangement — it would see an unordered bag of patches. The position embeddings are learned (not sinusoidal) and encode the 2D grid position of each patch.

Feed-Forward Network

x → Linear(1024, 4096) // expand 4× → QuickGELU() // activation → Linear(4096, 1024) // contract back

The 4× expansion ratio is standard in transformers. The FFN lets each token independently transform its representation.

Why all tokens attend to all?

Unlike the LLM's causal mask, the vision encoder uses full bidirectional attention. Every patch can see every other patch. This is critical — the bottom-right patch might need context from the top-left to understand the scene.

Attention matrix size: 577 × 577 = 332,929 entries per head × 16 heads. This is why vision token count matters — attention is O(n²).

Why layer -2 instead of the last layer?
The final layer of CLIP is heavily optimized for the contrastive image-text matching loss (projecting toward a single global embedding). This makes it less useful for spatial understanding. The second-to-last layer retains richer, more spatially detailed features that haven't been collapsed toward a single global vector yet.
// Feature extraction: features = hidden_states[-2] // (1, 577, 1024) visual_tokens = features[:, 1:] // (1, 576, 1024) // ↑ drop [CLS] at index 0
Why drop [CLS]? The [CLS] token is trained for global image-level representation (used in CLIP's contrastive loss). For VLMs, we want per-region spatial detail — the 576 patch tokens each represent a specific 14×14 region of the image.

Why a projector? CLIP and the LLM were trained separately with different embedding spaces. The projector learns to translate visual features into "pseudo-text" tokens the LLM can understand. It's the only new component trained in LLaVA's stage-1 pre-training.
LLaVA v1 vs v1.5: v1 used a single linear layer (1024→4096). v1.5 upgraded to a 2-layer MLP with GELU, which significantly improved performance. The extra layer gives more capacity to learn the vision→language mapping.

The LLM doesn't know the difference! After projection, visual tokens are indistinguishable in format from text tokens — they're just 4096-dim vectors. The LLM's self-attention naturally learns to cross-attend between visual and text tokens. This is the elegance of LLaVA's approach: no architectural changes to the LLM needed.
Training Strategy (Two Stages)

Stage 1: Pre-training (Feature Alignment)

Data: ~558K image-caption pairs (CC3M filtered)

What's trained: Only the MLP projector

Frozen: Vision encoder + LLM

Goal: Teach the projector to map CLIP features into the LLM's vocabulary space. The LLM learns to "read" image tokens as if they were text.

Stage 2: Fine-tuning (Instruction Tuning)

Data: ~665K instruction-following samples (multi-turn)

What's trained: MLP projector + LLM

Frozen: Vision encoder

Goal: Teach the full system to follow visual instructions — answer questions, describe images, reason about visual content.

Efficiency insight: In Stage 1, only ~20M parameters (the projector) are trained — less than 0.2% of the total model. This makes LLaVA remarkably cheap to train initially. Stage 2 unfreezes the LLM but keeps the vision encoder frozen, leveraging CLIP's pre-trained visual features.
