Deep Dive: How Patching Works VISION

The very first transformation — turning continuous pixel data into a sequence of discrete tokens

336 × 336 image with 24 × 24 patch grid
Hover over the grid to inspect individual patches

What is Patching?

Instead of feeding individual pixels (336×336 = 112,896 pixels!), we divide the image into a grid of non-overlapping patches. Each patch becomes one token.

1
Divide — Split the 336×336 image into a 24×24 grid of 14×14 pixel patches. That's 576 patches total.
2
Flatten each patch — Each 14×14×3 patch = 588 raw values
3
Project — A linear layer (Conv2d) maps 588 → 1024 dimensions. This is a learned projection, not hand-crafted features.
// The Conv2d trick: equivalent to the 3 steps above, but GPU-efficient Conv2d( in_channels = 3, // RGB input out_channels = 1024, // embedding dimension kernel_size = 14, // = patch_size stride = 14 // = patch_size (non-overlapping!) ) // kernel=stride means no overlap → each pixel belongs to exactly 1 patch
Why not individual pixels? Self-attention is O(n²). With 112,896 pixels that's ~12.7 billion attention entries per layer — completely infeasible. With 576 patches, it's only ~332K entries. Patching makes vision transformers tractable.

What if the image isn't 336×336? CLIPProcessor first resizes the shortest side to 336, then center-crops to 336×336. Aspect ratio information is lost (LLaVA-1.5-HD addresses this with multi-crop strategies).

// A single learnable 1024-dim vector cls_token = nn.Parameter(torch.randn(1024)) // Prepend to the patch sequence: cls_expanded = cls_token.unsqueeze(0).expand(B, 1, -1) sequence = cat([cls_expanded, patch_embeds], dim=1) // (1, 576, 1024) → (1, 577, 1024)
Purpose: The [CLS] token has no fixed meaning — through training, it learns to aggregate global image information via self-attention with all patches. CLIP uses it for the image-text contrastive loss. LLaVA drops it before projection, keeping only the 576 spatially-grounded patch tokens.
Position Embeddings
// Learnable embedding table: one vector per position pos_embed = nn.Embedding(577, 1024) // position_ids = [0, 1, 2, ..., 576] output = sequence + pos_embed(position_ids) // Element-wise addition, shape unchanged: (1, 577, 1024)
Why needed? Self-attention is permutation-invariant — without position info, the model can't tell if a patch is top-left or bottom-right. Position embeddings encode the 2D spatial layout. They are learned during CLIP pre-training (not sinusoidal).

What Do the 24 Layers Learn?
Why layer -2 (23rd) and not the last (24th)? The final layer's hidden states are heavily shaped by CLIP's contrastive loss — they're optimized to produce a single global vector (via [CLS]) for image-text matching. This "collapses" spatial information. Layer -2 still has rich semantic understanding but retains more spatial detail in each patch token, which is exactly what the LLM needs to answer spatial questions like "What's in the top-left corner?"

// LLaVA v1.5 projector — applied identically to each of the 576 tokens class MultimodalProjector(nn.Module): def __init__(self): self.linear_1 = nn.Linear(1024, 4096) // vision_dim → llm_dim self.act = nn.GELU() self.linear_2 = nn.Linear(4096, 4096) // llm_dim → llm_dim def forward(self, x): // x: (576, 1024) return self.linear_2(self.act(self.linear_1(x))) // → (576, 4096)
It's token-wise! The projector applies the same MLP independently to each of the 576 patch tokens. There's no cross-token interaction here — that already happened in the transformer. Think of it as a "translator" that converts each vision-space vector into the LLM's language.
Total projector params: ~21M — a tiny fraction of the overall model (~13B). Yet this tiny bridge is what makes the whole system work. In Stage 1 pre-training, this is the only component being trained.

// The key insight: visual tokens replace the <image> placeholder input_ids = tokenize("A chat between... USER: <image>\nDescribe this image ASSISTANT:") // ↑ this gets replaced // Find where <image> token is: image_pos = find(input_ids, IMAGE_TOKEN_ID) // Build the embedding sequence: text_embeds = embed_table(input_ids) // all text → (seq_len, 4096) visual_embeds = projector(vision_features) // image → (576, 4096) // Splice visual tokens where <image> was: input_embeds = cat([ text_embeds[:image_pos], // tokens before <image> visual_embeds, // 576 visual tokens text_embeds[image_pos+1:] // tokens after <image> ], dim=0) // → (~612, 4096) — ready for the LLM's first transformer layer
The LLM treats visual tokens exactly like text tokens. Both are 4096-d vectors in the same embedding space. The LLM's self-attention naturally cross-attends between visual and text tokens — when generating "red", it can attend to the left-side patches; when generating "blue", to the right-side patches. No architectural modification to the LLM is needed.

Vision vs. Language Attention — Key Differences

Same mechanism, different masks and objectives

Vision Encoder (CLIP) Language Model (LLM)
Attention Type Bidirectional (full) — every token sees all others Causal (masked) — token can only see previous tokens
Why Image understanding needs full context (bottom patch might need top patch info) Autoregressive generation — can't peek at future tokens
Sequence Length 577 tokens (fixed for a given image resolution) Variable (576 visual + N text, up to context limit)
Architecture Pre-Norm (LayerNorm before attention/FFN) Pre-Norm (same style, with RMSNorm in LLaMA)
Positional Encoding Learned absolute (577 positions) RoPE (rotary, relative)
Hidden Dim 1024 4096 (Vicuna-13B) or 5120 (13B)
